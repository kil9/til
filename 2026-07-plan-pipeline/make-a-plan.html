/make-a-plan

작업 계획을 수립하고 PLAN_XXX.md 마스터 문서를 만든 뒤 태스크를 순서대로 진행한다.

BashReadEditWriteGlobGrepAskUserQuestion

작업 설명: $ARGUMENTS

조사로 메워지지 않는 디테일(문제 정의, 목표·비목표, 접근 방식, 작업 범위, 제약, 테스트 전략, 우선순위 등)은 의문이 없어질 때까지 집요하게 AskUserQuestion 으로 사용자를 인터뷰한다. 추측으로 채우지 않고, 줄글 자유 입력도 쓰지 않는다.

확정된 내용으로 현재 디렉토리에 PLAN_<slug>.md(소문자 케밥케이스 2\~4단어) 를 작성한다. 세션 리셋 후에도 이 파일만으로 재개 가능할 만큼 상세히 적고, 배경/문제/목표/비목표/제약, 접근 방식과 근거, 태스크 목록, 검증 방법, 마지막 갱신 날짜를 포함한다. 같은 파일이 있으면 덮어쓸지 확인한다.

넘버링 규칙 (M / T / I).

·M-N 마일스톤: 태스크를 묶는 상위 단위 (M-1, M-2 …). 규모가 작아 묶을 필요가 없으면 마일스톤 없이 태스크만 나열해도 된다.
·T-N 태스크: 마일스톤 아래의 실행 단위 (T-1, T-2 …), 상태 [ ] TODO/[→] IN_PROGRESS/[x] DONE. 플랜 전체에서 유일하게 증가하는 전역 카운터(중복·재사용 금지, 마일스톤이 바뀌어도 리셋하지 않는다).
·I-N 아이디어: 보류 항목 (I-1, I-2 …), 별도 카운터.

아직 착수하지 않을 후보 아이디어가 있으면 태스크(T-N)로 넣지 말고 별도 ## 아이디어 (보류) 섹션에 I-N 으로 적는다(follow-plan 진행 대상 아님, 착수 지시 시 상세 인터뷰 후 태스크로 편입). 아이디어 추가·관리는 /add-idea 스킬을 쓴다.

PLAN 경로와 태스크 목록을 보고하고 "T-1부터 진행할까요?" 로 확인받는다. 확인되면 태스크를 하나씩 순서대로 진행한다: 상태를 [→] IN_PROGRESS 로 바꾸고 → 구현 → 검증 → [x] DONE 및 진행 로그·날짜 갱신 → /commit 으로 커밋 → 다음 태스크. 검증 실패 시 원인 파악 후 재구현하며, PLAN 은 항상 실제 상태를 반영한다.
