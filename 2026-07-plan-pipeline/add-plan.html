/add-plan

간단한 조사만 거쳐 PLAN.md에 새 태스크 항목을 추가한다. 작업은 시작하지 않는다. PLAN.md가 없으면 /make-a-plan을 자동 수행한다.

BashReadEditWriteGlobGrepAskUserQuestionSkill

추가할 태스크: $ARGUMENTS

"(아이디어)" 접두 항목은 이 스킬로 처리하지 않는다. $ARGUMENTS 가 "(아이디어)"로 시작하면(또는 사용자가 아직 착수하지 말고 보류만 원하면) 태스크로 만들지 말고 Skill("add-idea") 로 넘겨 PLAN.md 의 "아이디어 (보류)" 섹션에 I-N 으로 기록한다. 인터뷰·구현도 하지 않는다.

현재 디렉토리에 PLAN.md 가 없으면 Skill("make-a-plan") 으로 먼저 만든 뒤 이어서 진행한다.

항목을 제대로 서술할 만큼만 코드베이스를 가볍게 조사한다(관련 파일·선행 태스크 확인 정도). 구현은 하지 않는다.

애매한 점(범위, 완료 조건, 의존 관계, 접근 방식 등)이 하나라도 남으면 모두 해소될 때까지 AskUserQuestion 으로 집요하게 사용자를 인터뷰한다. 줄글 자유 입력은 쓰지 않는다.

넘버링 규칙 (M / T / I). 플랜의 항목은 세 계열로 번호를 매긴다.

·M-N 마일스톤: 태스크를 묶는 상위 단위 (M-1, M-2 …).
·T-N 태스크: 마일스톤 아래의 실행 단위 (T-1, T-2 …). 플랜 전체에서 유일하게 증가하는 전역 카운터로, 현재 존재하는 가장 큰 T-N 다음 번호를 부여한다(중복·재사용 금지, 마일스톤이 바뀌어도 리셋하지 않는다).
·I-N 아이디어: 보류 항목 (I-1, I-2 …), 별도 카운터. /add-idea 가 관리.

새 태스크는 이 규칙에 따라 적절한 마일스톤 아래 "다음 할 일"에 해당하는 위치 맨 아래에 T-<다음 번호> 로 추가한다. 마일스톤 구분이 없는 기존 플랜이면 마일스톤은 강제로 만들지 말고 기존 포맷을 존중하되, 태스크 번호는 T-N 을 쓴다. 원문 ID 표기가 표준(M-/T-/I-)과 다르면(예: T1, NU-3) 그 표기를 그대로 이어서 붙인다. 그 외 기존 내용은 수정하지 않는다.

마지막으로 추가된 항목의 번호·제목을 알리고 끝낸다. 구현·커밋은 하지 않는다.
