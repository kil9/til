Cleanup Plan

완료된 작업의 세부 구현 기록을 지우고, 재개에 필요한 미완료 작업과 결정 사항은 보존한다. PLAN 파일만 수정한다 — 코드 변경·테스트는 하지 않고, 커밋은 사용자가 명시적으로 요청할 때만 한다.

절차

1.대상 PLAN 파일을 고른다.
·현재 디렉토리에 PLAN.md가 있으면 그것을 사용한다.
·없으면 PLAN_*.md 중 가장 최근 수정된 파일을 사용한다.
2.파일 전체를 읽고 태스크 ID 체계를 파악한다. 표준은 M-N(마일스톤)/T-N(태스크)/I-N(아이디어)이며, T1·NU-3 같은 커스텀 표기도 있을 수 있다. 원문에 적힌 ID 표기를 그대로 보존한다(표준으로 바꾸지 않는다).
3.완료 태스크만 정리 대상으로 삼는다.
·완료로 보는 신호: [x], DONE, 완료, 커밋/검증까지 끝났다고 명시된 태스크.
·정리하지 않는 신호: [ ], [→], TODO, IN_PROGRESS, 진행 중, blocked, 보류, 아이디어.
·완료 여부가 조금이라도 애매하면 원문을 보존한다.
4.모든 완료 태스크를 빠짐없이 한 줄 요약으로 치환한다.
·완료 태스크의 하위 체크리스트, 상세 설명, 검증 로그, 커밋 로그, 리뷰 메모는 삭제한다.
·남기는 형식은 기존 PLAN 스타일에 맞추되, 기본은 다음 형태다.
- [x] T-12: 짧은 요약
·기존 제목이 충분히 짧으면 제목을 요약으로 사용하고, 길거나 구현 세부가 섞여 있으면 결과 중심으로 새로 쓴다.
·새로 쓰는 범위 표기 ~는 Markdown 규칙에 맞게 \~로 escape한다.
5.살아 있는 컨텍스트는 보존한다.
·문제 정의, 목표/비목표, 제약, 설계 결정, 리스크, 미완료 태스크, 보류 아이디어는 삭제하지 않는다.
·단, 완료 태스크 내부에만 있던 세부 결정이 이후 작업에 필요하면 태스크 밖의 적절한 "디자인 결정" 또는 "리스크/주의" 섹션으로 옮긴 뒤 압축한다.
6.변경 후 결과를 검증한다.
·삭제된 내용이 완료 태스크 세부 기록에 한정되는지 확인한다.
·미완료/진행 중/보류 항목의 본문이 바뀌지 않았는지 확인한다.
·마지막에 정리한 파일, 압축한 태스크 수, 보존한 미완료 태스크 수를 보고한다.
