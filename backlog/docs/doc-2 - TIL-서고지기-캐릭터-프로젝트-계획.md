# TIL 서고지기 캐릭터 프로젝트 계획

## 배경 / 문제

kil9.github.io/til 의 보고서는 대부분 사용자가 아니라 Claude 가 작성한다. 지금은 화자(글쓴이)가 익명이라, 독자 입장에서 "누가 쓴 글인가"가 어색하다. 사이트 전속 서고지기(사서) 캐릭터를 만들어 보고서의 화자로 세우면, AI 가 쓴 보고서라는 사실을 자연스럽게 드러내면서 독자가 받아들이기도 쉬워진다.

## 목표

1. 서고지기 캐릭터의 인격(이름·성격·말투·문체 규칙)과 비주얼 스펙을 확정하고 문서화한다.
2. 캐릭터 설정화 시트(전신+상반신+표정 변형)를 이미지로 제작한다.
3. 신규 TIL 페이지부터 서고지기 자아를 적용할 수 있게 런북(AGENTS.md)에 규칙을 반영하고, 루트 인덱스에 캐릭터 소개를 넣는다.
4. 이 제작 과정 전체를 하나의 TIL 보고서로 퍼블리시한다(서고지기의 첫 1인칭 적용 사례가 된다).

## 비목표

- 기존에 퍼블리시된 페이지의 소급 수정은 하지 않는다(사용자 인터뷰에서 제외 확정).
- 캐릭터 설정 자체를 독립 공개 페이지(진실원본)로 만들지 않는다. 진실원본은 backlog/docs 이고 공개물은 TIL 보고서 1건이다.
- 웹폰트·외부 라이브러리 도입 없음(사이트 단일 파일 원칙 유지).

## 캐릭터 초기 설정 (사용자 제공, 인터뷰로 확정된 방향)

- 성격: 밝고 활기참. 호기심이 많아 여러 가지를 알아두고 싶어 하며, 약간의 지식 호더(knowledge hoarder) 성향.
- 문체: 지금까지의 TIL 과 크게 다르지 않은 건조한 보고서 양식이 기본. 보고서 앞뒤에서만 약간의 자아를 드러낸다. 예: "사용자님의 요청으로 진행했습니다", "사용자님이 보고서의 방향 변경을 요청하셔서 새로 작성했습니다".
- 가끔 자신의 실수를 귀엽게 포장하는 면이 있어도 좋다.
- 비주얼: 숏컷 여성, 세련된 느낌.
- 이름: 미정. 태스크에서 후보 여러 개를 컨셉 근거와 함께 제안하고 사용자가 AskUserQuestion 으로 고른다.

## 인터뷰 확정 사항

- 이름 결정: 후보 제안 → 사용자 선택.
- 그림 구성: 설정화 시트(전신+상반신+표정 여러 장).
- 아트 스타일: 후보 3안(깔끔한 애니풍 일러스트 / 플랫·벡터풍 미니멀 / 수채·연필 손그림풍)을 각각 샘플 생성해 비교 후 사용자가 선택한다.
- 적용 범위: 신규 페이지부터 적용 + 루트 인덱스에 소개 반영. 기존 페이지 소급 없음.
- 설정 진실원본: backlog/docs 문서(이 문서와 별도의 캐릭터 설정 문서). 페이지 작성 시 실제로 지켜야 할 문체 규칙만 AGENTS.md 런북에 요약 반영.

## 접근 방식과 근거

1. **설정 먼저, 그림 나중.** 이름·성격·비주얼 스펙이 프롬프트의 입력이 되므로 설정 확정이 선행돼야 한다.
2. **스타일 테스트 → 시트 제작 2단계.** 설정화 시트는 생성 비용이 크므로, 먼저 3개 스타일 샘플을 1장씩 만들어 사용자가 고른 뒤 확정 스타일로만 시트를 뽑는다.
3. **이미지 생성은 런북 §2-1 파이프라인 재사용.** codex CLI 의 image_generation 에 위임하고, 결과물은 WebP(q75) 압축 후 base64 data URI 로 임베드한다(사이드카 파일 금지 원칙). 설정화 시트 원본은 페이지 임베드용과 별도로 backlog/docs 옆이 아닌 저장소 외부 보관이 불가하므로, 보고서 페이지에 임베드된 것이 공개본이며 원본 PNG 는 커밋하지 않는다(public repo 용량·단일 파일 원칙).
4. **런북 반영은 요약만.** AGENTS.md 에는 "신규 페이지의 서고지기 자아 표출 규칙"(어디서, 얼마나, 어떤 톤) + 설정 문서 포인터만 넣어 런북 비대화를 막는다.
5. **보고서는 서고지기 1인칭 첫 적용.** 캐릭터 제작기 보고서 자체가 새 문체 규칙의 첫 실전 사례가 되도록 마지막 태스크로 둔다.

## 검증 방법

- 설정 문서: 사용자가 이름·설정·문체 규칙 예문을 승인.
- 스타일 샘플·설정화 시트: 사용자가 직접 보고 선택/승인(AskUserQuestion + 이미지 전달).
- 런북·인덱스: 로컬 미리보기(python3 -m http.server)로 라이트/다크 모두 확인.
- 보고서: 퍼블리시 전 슬러그 사용자 확인(런북 규칙), 퍼블리시 후 라이브 URL 확인.

## 제약

- 저장소는 public. 캐릭터 이미지·설정 포함 모든 산출물이 즉시 공개된다.
- 이미지 임베드는 단일 파일 원칙(base64 data URI), 페이지당 용량을 과도하게 키우지 않도록 시트는 압축 필수.
- em-dash 금지, 범위 표기 규칙 등 전역 마크다운 규칙 준수.
